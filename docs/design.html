의도를 받아 브라우저를 검증하는
QA 에이전트의 런타임

자연어 의도를 실제 브라우저 동작으로 옮기고, 수집한 증거로 결과를 판정한다. 하네스는 LLM 자체가 아니라 그 주변 설계 — 무엇을 컨텍스트로 넣고, 어떻게 실행하고, 무엇으로 판정하고, 언제 사람을 부를지 — 를 가리킨다.

런타임 TypeScript / Node 드라이버 Chrome DevTools MCP 모델·드라이버 비종속 QA = 첫 응용
01

설계 제약

하네스가 따르는 네 가지 제약.

모델·드라이버 비종속

LLM도 브라우저 드라이버도 인터페이스 뒤에 둔다. 기본 구현을 갈아끼울 수 있다.

결정적 재생

회귀 실행은 LLM 없이 굳힌 시나리오를 재생한다. (→ §04)

패턴 / 데이터 경계

파이프라인·인터페이스는 코어, 도메인 데이터·커넥터는 플러그인으로 분리한다.

증거 기반 판정

화면이 아니라 실행·화면·로직 3층 증거로 판정한다. (→ §06)

실행의 본체는 파이프라인이며, 루프는 처음 보는 앱을 탐색하는 단계(§04)에서만 쓴다.

02

요구 → 구조

QA가 요구하는 것에서 구성요소를 거꾸로 도출한다.

QA가 필요로 하는 것그래서 필요한 메커니즘
대상 앱을 전혀 모름컨텍스트 주입 — 의도·도메인을 grounding
의도를 실행 단계로 풀어야 함시나리오 생성(Plan)
사람처럼 조작해야 함드라이버 (브라우저 도구)
처음 보는 앱이라 경로를 미리 못 짬탐색 루프 — 보고·행동·적응 (여기만 루프)
페이지가 늦거나 동적임단계별 적응 재시도 (robustness)
증거로 옳고 그름을 판정증거 수집 + Critic
회귀 때 LLM 비용을 안 내야 함성공 경로 굳히기(freeze) → 재생
위험한 순간엔 사람 판단체크포인트(사람 게이트)

이 표가 아키텍처가 된다. 대부분은 순서가 정해진 파이프라인이고, 루프는 “처음 보는 앱”에서만 요구된다.

04

탐색과 재생

루프가 정당하게 필요한 단 하나의 자리.

파이프라인에서 불확실성이 있는 지점은 하나다 — 처음 보는 앱. DOM을 모르면 “장바구니에 담아”를 어디 클릭할지 미리 짤 수 없다. 그래서 첫 실행은 탐색이다: 보고 → 행동하고 → 바뀐 화면을 보고 → 다음을 정한다.

탐색 · DISCOVER 의도 + 모르는 앱 → ↻ 관찰 · 행동 · 적응 → 경로 발견 (LLM 루프)
↓ 성공하면 결정적 시나리오로 굳힘 (freeze) — 루프가 여기서 소멸
재생 · REPLAY 굳힌 시나리오 → 그대로 재생 (LLM·루프 없음 · 결정적)
탐색이 성공하면 경로를 굳히고(freeze), 이후 실행은 결정적 시나리오를 재생한다. 재생 경로에는 LLM이 없으므로 회귀 비용과 flaky가 함께 줄어든다.

LLM이 다시 등판하는 건 두 경우뿐이다 — 새 시나리오를 탐색할 때, 그리고 굳힌 경로가 UI 변경으로 깨져 복구(self-heal)할 때. 그 외 모든 실행은 결정적 재생이다.

탐색적(exploratory) 모드 — 새 이슈가 안 나올 때까지 반복 — 은 선택적 확장이다.

05

확장점

파이프라인 각 단계는 교체 가능한 인터페이스 — “패턴은 코어, 데이터는 플러그인”.

// 파이프라인 단계 ↔ 확장점 interface ContextProvider { provide(task): Context } // NL · git diff · 티켓 · 문서RAG interface Planner { plan(ctx): Scenario } // 의도 → 단계 interface Driver { goto·click·type·observe() } // 기본 ChromeDevToolsMCP / 교체 Playwright interface SkillStore { resolve(name): Skill } // 재사용 플로우 · freeze / replay interface Critic { judge(evidence): Verdict } // 단언 | baseline | LLM interface Reporter { emit(result) } // console · json · (회사: Jira)

core 가 들고 있는 것

파이프라인 오케스트레이션, 기본 드라이버, 탐색·시나리오 생성 프롬프트, 3층 증거 스키마, freeze 로직.

plugin 으로 꽂는 것

환경별 ContextProvider·Reporter 구현. 회사 버전은 이 둘만 갈아끼우면 끝 — 코어 변경 0.

경계 = 패턴 vs 데이터. 오픈소스로 나가는 건 파이프라인·인터페이스·프롬프트(패턴). 도메인 코퍼스·내부 커넥터·시나리오 자산(데이터)은 사내에 남는다.
06

증거와 판정

Critic은 단정하기 전에 관찰 가능한 사실을 세 층위로 받는다.

화면만 보는 검증은 화면 단에서 멈춘다. Critic이 받는 증거는 실행·화면·로직 세 층이라, 화면상으론 멀쩡한데 실제론 실패한 경우를 잡는다.

Evidence = { execution: { clicked: true, navigated: true, blocked: false }, // 실행층 perception: { screenshot: "step3.png", layoutShift: 0 }, // 화면층 logic: { requests: [{ url: "/api/orders", status: 500 }], // 로직층 console: ["TypeError: orders is null"] } }
알려진-항목 억제(known-item suppression). 입력(변경 요약·티켓)에 “이건 의도적으로 미적용”이라 적혀 있으면 Critic이 버그가 아닌 것으로 분류한다. AI QA의 고질병인 오탐을 입력 맥락으로 누른다.
07

사용 면

헤드리스 CLI와 시각적 리플레이 — 같은 런타임의 두 표면.

CLI · CI

한 줄로 시나리오를 돌리고 구조화된 리포트를 떨군다. GitHub Action에 그대로 물려 PR 게이트로 쓴다.

zsh — qa-agent
$ npx qa-agent run "checkout: 담기 → 결제 → 주문확인"
✓ context assembled — 3 docs · 2 skills
✓ scenario planned — 5 steps
▶ step 3/5 click "Pay" POST /api/pay 200
▶ step 5/5 verify order GET /api/orders 500
✗ 1 issue — order not shown · evidence captured
  report → ./qa-report.json · exit 1

시각적 리플레이

같은 실행을 로컬 UI로 따라간다. 단계 타임라인 + 라이브 뷰 + 3층 증거. 비개발자 검토와 디버깅용.

입력 소스 · 통합

입력은 모두 ContextProvider 한 인터페이스로 들어온다 — ① 자연어 ② git diff / PR ③ 티켓 ④ 저장된 시나리오. 사내 버전은 그 위에 커넥터 두 개만 구현한다.

class EpilogueProvider implements ContextProvider { /* 작업 요약 → Context */ } class JiraReporter implements Reporter { /* 이슈 → 버그 티켓 초안 */ } createHarness({ context: new EpilogueProvider(), reporter: new JiraReporter() })
08

로드맵

파이프라인 하나부터. 레이어는 필요할 때 얹는다.

v0

최소 파이프라인 — “MCP가 도나”

NL 1개 → 드라이버 실행 → 3층 증거 → Critic(LLM+텍스트 단언) → JSON. 탐색·freeze·게이트 제외.

v1

탐색 + 굳히기

처음 보는 앱 탐색 루프 → 성공 경로 freeze → 결정적 재생. self-heal 재시도.

v2

입력 소스 + 사람 게이트

git diff·티켓 ContextProvider, 알려진-항목 억제, 위험 단계 사람 게이트, 시각적 리플레이.

v3

생태계 · 사내 적용

GitHub Action, baseline diff, 커넥터 SDK → 에필로그·Jira 사내 버전.

모노레포 packages/harness(코어) + packages/qa(앱). 라이선스 Apache-2.0. 이 문서가 곧 README의 뼈대.

QA Agent Harness · 설계 문서 — (가칭) Probe. 코드·화면은 설계 설명용 예시이며 구현 과정에서 변경될 수 있음.
