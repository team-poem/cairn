자연어로 적으면 테스트 케이스를 대신 검증하는 QA 에이전트

“로그인하고 첫 상품을 장바구니에 담아”처럼 평소 말로 적으면, AI가 브라우저에서 그 경로를 한 번 찾아 저장한다. 그다음부터는 AI 없이 똑같이 다시 돌려 — 빠르고 매번 같은 결과 — 화면·네트워크·콘솔 증거로 합격/불합격을 따진다. cairn은 AI 자체가 아니라 그 주위를 감싸는 엔진이다.

엔진 cairn-engine@1.0.0 · npm 런타임 TypeScript / Node 기본 드라이버 Chrome DevTools MCP 모델·드라이버 비종속 · MIT
01

설계 원칙

엔진이 지키는 — 코드와 PR 훅으로 강제되는 — 다섯 제약.

모델·드라이버 비종속

AI도 브라우저 도구도 인터페이스 뒤에 둔다. 기본은 Chrome DevTools MCP, 교체 가능(Playwright 등).

재생은 AI 없이

반복 실행은 AI 없이 저장한 시나리오를 그대로 돌린다 → 매번 같은 결과. (→ §04)

패턴 / 데이터 경계

파이프라인·인터페이스는 코어, 도메인 데이터·커넥터·시나리오 자산은 제품에 남긴다.

증거 기반 판정

“화면이 맞아 보였다”가 아니라 실행·화면·로직 3층 증거로 판정. (→ §06)

다섯째 — 확장은 인터페이스(포트)로만. 단계 안에 분기를 박지 않는다. 그리고 루프(관찰·행동·적응)는 ‘처음 보는 앱을 탐색’할 때만 쓴다.

02

요구 → 구조

QA가 실제로 필요로 하는 것에서 부품을 거꾸로 끌어냈다.

QA가 필요로 하는 것그래서 필요한 부품
대상 앱을 전혀 모름컨텍스트 주입 — 의도를 실제 화면에 맞춤
의도를 실행 단계로 풀어야 함시나리오 생성(Plan)
사람처럼 눌러야 함드라이버 (브라우저 도구)
처음 보는 앱이라 경로를 미리 못 짬탐색 루프 — 보고·누르고·다시 보고 (여기만 루프)
페이지가 늦거나 동적임로딩이 끝날 때까지 기다리는 자동 대기
증거로 옳고 그름을 판정증거 수집 + Critic
반복 실행 땐 AI 비용을 안 내야 함성공 경로 저장(freeze) → 재생
성공 기준이 제품마다 다름제품이 정의하는 custom 단언

이 표가 그대로 아키텍처가 됐다. 대부분은 순서가 정해진 파이프라인이고, 루프는 “처음 보는 앱”에서만 필요하다.

코드는 코어(core/ — 핵심 규칙)와 꽂는 부품(adapters/ — 실제 구현)으로 갈라져 있다.

컨텍스트를 모으고 → 할 일을 단계로 풀고 → 브라우저를 누르고 → 따지고 → 결과를 낸다. Execute엔 페이지 로딩이 끝날 때까지(네트워크가 조용해질 때까지) 기다려 주는 자동 대기가 있어, 늦게 뜨는 화면 때문에 잘못 실패하지 않는다.

04

핵심 베팅 · 탐색과 재생

AI 탐색은 딱 한 번만 비싸게, 그다음 반복은 AI 없이 공짜.

불확실한 곳은 하나뿐이다 — 처음 보는 앱. 화면을 모르면 “장바구니에 담아”를 어디 누를지 미리 짤 수 없다. 그래서 첫 실행은 탐색이다: 보고 → 누르고 → 바뀐 화면을 보고 → 다음을 정한다.

🔎
discover
AI · 1회 · ~$0.5
→
🧊
freeze
JSON 저장
→
▶
replay
AI 0 · 매번 같음 · $0
→
🩹
self-heal
AI · 깨질 때만
탐색 · DISCOVER 하고 싶은 일 + 모르는 앱 → ↻ 보고 · 누르고 · 다시 보고 → 경로 발견 (AI가 한 번 도는 단계)
↓ 성공한 경로를 파일로 저장(freeze) — AI 루프는 여기서 끝
재생 · REPLAY 저장한 경로 → 그대로 다시 실행 (AI 없음 · 매번 같은 결과 · ~4초 · $0)
AI의 유연함은 얻되, 매 실행마다 드는 AI의 비용·불안정함은 버린다. 다시 돌릴 때 AI가 없으니 비용도, 매번 결과가 달라지는 불안정함(flaky)도 함께 사라진다. Slack 글이 “AI 에이전트는 CI 자동검사엔 쓰지 마라”고 한 그 자리를, cairn은 ‘탐색은 한 번 / 반복은 AI 없이’로 나눠서 차지한다.

AI가 다시 나서는 건 둘뿐 — 새 경로 탐색, 그리고 저장한 경로가 UI 변경으로 깨져 고칠(self-heal) 때. self-heal은 늘 도는 게 아니라 깨졌을 때만 가끔 도는 예외다.

05

확장점 — 제품이 정의한다

“우리가 정한 것만 흐른다”가 아니다. 모든 층이 교체·확장 가능하다.

// 7개 포트 — 환경별 동작을 꽂는 자리 interface ContextProvider { provide(task): Context } // NL · git diff · 티켓 · RAG interface Planner { plan(ctx): Scenario } // 의도 → 단계 interface Driver { goto · click · type · locate · // 기본 Chrome DevTools MCP select · observe · close() } // 교체 Playwright 등 interface SkillStore { resolve(name): Skill } // freeze / replay 보관 interface Critic { judge(evidence, asserts) } // 기계적 | LLM | 제품 정의 interface Reporter { emit(result) } // console · json · 임의 트래커 interface LlmClient { complete(prompt) } // Claude Code · Anthropic · BYO

닫힌 데이터마저 열려있다 — 제품이 성공 기준과 인터랙션을 직접 정의한다:

await runScenario(scenario, { custom: { "cart-has": (p, ev) => ev.logic.requests.some(r => r.url.includes(p.path) && r.status === 200) }, actions: { "drag-slider": async (driver, p) => { /* 제품 고유 인터랙션 */ } }, })
경계 = 우리는 루프·포트·좋은 기본값, 제품은 specifics(액션·단언·로케이터·컨텍스트). 오픈소스로 나가는 건 패턴(엔진), 도메인 데이터·시나리오 자산은 제품에 남는다.
06

증거와 판정

Critic은 “화면이 맞아 보였다”가 아니라 관찰 가능한 사실 3층으로 따진다.

화면만 보는 검증은 화면에서 멈춘다. cairn은 실행·화면·로직 세 층을 받아, 화면상으론 멀쩡한데 실제론 실패한 경우(주문 API가 500인데 “주문 완료”처럼 보임)를 잡는다.

Evidence = { execution: { actions, navigated: true, finalUrl, blocked: false }, // 실행층 perception: { screenshot: "data:image/png;…" }, // 화면층 (스텝별) logic: { requests: [{ url: "/api/orders", status: 500 }], // 로직층 console: [{ type: "error", text: "orders is null" }] } }

단언은 AI의 추측이 아니라 실제 일어난 일에 맞춘다 — 정말 페이지를 옮겼으면 올바른 목적지(host+path)까지 확인하고, 항상 실패 요청(4xx/5xx, favicon 같은 무해한 건 제외) 없음을 본다. “성공이 뭔지”는 제품이 custom 단언으로, 애매한 목표는 expect(AI 판정, 선택)로.

다음 단계: 입력(변경 요약·티켓)에 “이건 의도적 미적용”이라 적히면 버그가 아닌 것으로 거르는 ‘알려진-항목 억제’ — AI QA 고질병인 오탐을 입력 맥락으로 누른다(설계 중, 미구현).

07

제품 형태

하나의 엔진, 세 입구 — 임베드(라이브러리) · CLI/CI · 데스크탑 앱.

① 임베드 + CLI · CI (지금)

엔진을 import하거나, cairn discover로 한 번 발견 → cairn replay를 매 PR에서 돌린다. 재생은 AI도 API 키도 필요 없다.

zsh — cairn
$ cairn discover "로그인하고 첫 상품을 담아 카트를 연다" --url shop.example --freeze cart.json
→ 6 steps · goto → type("Username") → click("Login") → click("Add to cart") → goto(/cart)
frozen → cart.json
$ cairn replay cart.json # 결정적, AI 0
✓ navigated → /cart.html  ✓ no-failed-requests  ✓ pass · exit 0

② 데스크탑 QA 앱 (다음 · 프로젝트 2)

엔진은 표시에 필요한 seam을 이미 다 노출한다 — onStep(타임라인)·screenshots(스텝별 PNG)·signal(Stop)·onHeal(노후 알림). 그 위의 UI는 엔진에 넣지 않고 별도 데스크탑 앱이 cairn-engine을 install해 그린다 — 로그인된 브라우저 세션 위에서, 비개발자도 보는 시각 리플레이.
